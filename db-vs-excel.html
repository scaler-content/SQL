📊Excel
Searching for Ankit's rides
You scroll through 10,000 rows manually, or write a VLOOKUP that crashes when it hits 50,000 rows.
🐌 Slow. Gets worse as data grows.
Multiple captains updating
Only one person can edit the file at a time. Two simultaneous updates → one person's data gets silently overwritten.
❌ Last to save wins. Data lost forever.
Connecting rider to rides
You'd manually copy the rider's name into every ride row, or write fragile VLOOKUP formulas. Update a name → update every row.
😰 Manual work. Breaks easily.
Rapido's ride history
Excel maxes out at ~1 million rows per sheet. Rapido does crores of rides every month across India.
❌ File crashes. Can't scale.
Protecting captain earnings
Anyone with the file can see and edit everything — including payouts — with no audit trail.
⚠️ No row-level control at all.
VS
🗄️Database
Searching for Ankit's rides
The database uses an index — like a book's index page — to jump straight to the right rows. No scrolling, no crashing, no manual work.
⚡ Milliseconds. Even with 10 crore rows.
Multiple captains updating
Databases handle thousands of simultaneous updates. Each is isolated — no conflicts, no overwriting.
✅ Thousands at once. No data loss.
Connecting rider to rides
The database links tables automatically using a Foreign Key. Update the rider's name once — it reflects everywhere, instantly, forever. No manual copy-pasting required.
✅ One source of truth. Zero duplication.
Rapido's ride history
PostgreSQL and MySQL handle billions of rows. Query speed stays fast due to indexing and query planning.
✅ Billions of rows. Still fast.
Protecting captain earnings
Role-based access: riders see their own rides. Payroll sees earnings. Admins see everything. Every change logged.
✅ Row-level security + full audit trail.
