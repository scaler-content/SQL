Transactions
All-or-nothing saves
Before any query runs, the DBMS checks: is this user allowed? A rider sees only their rides. A captain sees only theirs. The CEO sees everything. Permissions work at the row and column level — very precise.
When you ask the database a question — like "show me all rides in Bangalore today" — the DBMS reads it, creates a plan to find that data as fast as possible, and returns the answer. Like Google Maps finding the fastest route — but for data.
The DBMS keeps a running log of every change. If a server crashes at 3am, Rapido's data is restored from backup with zero rides lost. Fully automatic — you never have to think about it.
When a ride is booked: (1) create ride record, (2) charge rider, (3) notify captain. All three must succeed together. If step 2 fails, step 1 is also undone. No partial saves. No money deducted without a ride.
